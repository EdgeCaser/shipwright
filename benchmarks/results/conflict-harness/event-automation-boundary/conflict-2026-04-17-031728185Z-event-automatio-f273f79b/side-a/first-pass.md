# Phase 1 PRD: Event Management Automation Platform

## Decision Frame

**What decision does this PRD enable?**
Whether to fund and staff a Phase 1 build of an event management automation platform that combines a WhatsApp-based client assistant, an internal operations console, and a human-approval gate for all pricing commitments — with LLM usage constrained to strictly bounded, non-deterministic-safe tasks only.

**Decision owner:** Product leadership and engineering leadership jointly.

**Decision deadline:** Must be made before engineering sprint planning for the next cycle.

**What a YES means:** Allocate engineering resources to build the four Phase 1 components described below, accept the architectural constraint that LLMs are prohibited from autonomous pricing or contract commitment, and staff a human reviewer role for approval workflows.

**What a NO means:** Defer or descope — either delay until operational cost of human approval gates is validated against throughput requirements, or reduce scope to the internal console only.

---

## Phase 1 Scope

### Component 1: WhatsApp Client Assistant
- Responds to inbound event inquiry messages (venue type, date, guest count, general budget range)
- Collects structured intake data and stores to a central event record
- Escalates ambiguous or high-stakes inputs to human operators via the internal console
- **LLM boundary:** LLM may interpret free-text intake fields and generate clarifying questions. LLM may NOT quote prices, confirm availability, or make scheduling commitments. All such outputs must be routed to a human operator for approval before being sent to the client.

### Component 2: Internal Operations Console
- Displays all active event inquiries, their current status, and pending human-approval items
- Allows operators to approve, modify, or reject LLM-drafted responses before delivery
- Surfaces pricing recommendations generated by a deterministic pricing engine (rules-based, not LLM-generated)
- Provides audit log of all approvals, rejections, and modifications

### Component 3: Human Approval Gate for Pricing Commitments
- Any outbound message containing a price, deposit requirement, package inclusion, or contractual term requires explicit operator approval before dispatch
- Approval interface presents: the original client message, the LLM-drafted response, the pricing engine's recommended figure, and a free-text override field
- SLA target for approval response is defined operationally, not enforced by the system in Phase 1
- Unapproved messages are held in a pending queue, not silently dropped

### Component 4: Deterministic LLM Boundary Enforcement
- A routing layer classifies every LLM output before dispatch: SAFE (informational, no commitment), REVIEW REQUIRED (any pricing, availability, or contract language detected), or BLOCK (output cannot be repaired and must not be sent)
- Classification is rule-based and deterministic — it does not use an LLM to decide if an LLM output is safe
- Logging captures the full LLM output, the classification result, and operator disposition for every message

## Unknowns & Evidence Gaps

1. **Operator throughput capacity.** It is unknown how many pricing approval requests a single operator can handle per hour without becoming a bottleneck. If event inquiry volume spikes, the human gate may become the system's constraint. No volume projections are available in this phase.

2. **WhatsApp Business API access and policy constraints.** The platform requires access to the WhatsApp Business API. Approval timelines, template message restrictions, and conversation window policies (24-hour rule) may constrain the assistant's ability to follow up on stale inquiries. These constraints are not yet characterized.

3. **Pricing engine data dependencies.** The deterministic pricing engine requires a defined ruleset and input data (venue capacity, date, services requested). It is unknown whether this data is currently structured and accessible, or whether a data modeling effort must precede Phase 1.

4. **Client tolerance for response latency.** Human approval introduces latency between client inquiry and response. It is unknown whether the target client segment (event organizers) expects near-instant replies or accepts a multi-hour turnaround. This materially affects whether the human gate is viable without an explicit expectation-setting mechanism.

5. **LLM output classification false-positive rate.** The REVIEW REQUIRED classifier will flag outputs for human review. If tuned conservatively, it may send nearly all outputs to the approval queue, negating automation benefit. If tuned loosely, it may miss pricing language. The acceptable false-positive/false-negative tradeoff has not been defined.

## Pass/Fail Readiness

### Go criteria (all must be true)
- [ ] WhatsApp Business API access confirmed and timeline to production credentials established
- [ ] Pricing engine input schema defined and at least one source of pricing rule data identified
- [ ] At least one operator role committed to staffing the approval queue for Phase 1
- [ ] LLM boundary classification logic drafted and reviewed by an engineer — no open design questions on the routing layer
- [ ] Legal/compliance sign-off that LLM-generated informational responses (pre-approval) do not constitute binding representations under applicable contract law

### Stop criteria (any one blocks launch)
- WhatsApp API access cannot be obtained in Phase 1 timeline
- No operator capacity available — human gate cannot be staffed
- Pricing data is unavailable or unstructured, making the deterministic pricing engine infeasible
- Legal review identifies that WhatsApp message receipt constitutes offer acceptance before operator approval

## Recommended Next Artifact

**Operational Runbook Draft** — before engineering begins, the team should produce a one-page operator runbook describing exactly what a human reviewer sees, what decisions they are empowered to make, and what the escalation path is for edge cases. This artifact will expose gaps in the approval gate design that a PRD cannot surface, and will validate whether the operator role is feasibly staffed. It should precede the technical design document.

Secondary recommendation: a WhatsApp API feasibility spike (1–2 engineer-days) to confirm account provisioning timeline and template constraints before committing to Phase 1 scope.
