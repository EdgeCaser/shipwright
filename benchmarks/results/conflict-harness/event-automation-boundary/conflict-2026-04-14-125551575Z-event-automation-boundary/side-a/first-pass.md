# Phase 1 PRD: Event Management Automation Platform

## Decision Frame

**Problem Statement:** Event management companies spend 60-70% of operator time on repetitive coordination tasks — venue availability checks, vendor follow-ups, client Q&A, and quote assembly — that could be automated. However, pricing commitments and contract terms carry financial and legal risk that requires human judgment.

**Decision Being Made:** Should we build a Phase 1 platform consisting of (1) a WhatsApp-based client assistant, (2) an internal operator console, and (3) a human-approval gate for all pricing/commitment actions, with strict deterministic boundaries around LLM usage?

**Scope Boundary:** Phase 1 covers single-event coordination workflows only. Multi-event portfolio management, payment processing, and contract execution are explicitly out of scope.

**Target User Segments:**
- **Primary:** Small-to-mid event management firms (5-30 employees) handling 20-100 events/month
- **Secondary:** Solo event coordinators managing 5-15 events/month

**Success Criteria for Phase 1:**
- Reduce operator time-per-event by ≥30% for covered workflow steps
- Zero unapproved pricing commitments reach clients
- WhatsApp assistant handles ≥80% of routine client queries without operator intervention
- Operator NPS ≥40 after 4-week pilot

## Product Architecture

### Component 1: WhatsApp Client Assistant

**What it does:** Responds to client inquiries via WhatsApp Business API — venue availability, event timeline status, document requests, FAQ responses.

**LLM Boundary (Strict):**
- LLM is used ONLY for natural language understanding (intent classification) and response generation from approved templates
- All factual responses (dates, prices, availability) are retrieved from the deterministic database layer, never generated by the LLM
- LLM output is constrained to template-fill operations: the system selects a response template deterministically based on classified intent, and the LLM fills natural language around verified data points
- If intent classification confidence < 0.85, the query is routed to a human operator immediately
- The LLM NEVER generates pricing figures, contractual terms, availability claims, or capacity numbers

**Deterministic Layer:**
- Intent classifier maps to enumerated action codes (VENUE_AVAIL, TIMELINE_STATUS, DOC_REQUEST, FAQ, ESCALATE)
- Each action code triggers a deterministic data lookup + template response
- All responses include a machine-readable audit trail linking every data point to its source record

### Component 2: Internal Operator Console

**What it does:** Dashboard for operators to manage events, review AI-drafted communications, approve pricing, and monitor assistant performance.

**Key Features:**
- Queue of pending approval items (pricing quotes, custom requests, escalated queries)
- Event timeline view with automated status tracking
- Assistant performance metrics (resolution rate, escalation rate, response time)
- Override controls: operator can correct any assistant response and the correction feeds back into template improvement
- Audit log of all assistant interactions with client, showing data sources for each response

### Component 3: Human Approval Gate

**What it does:** Enforces mandatory human review before any pricing commitment, discount, or contractual term reaches a client.

**Gate Rules (Deterministic, Not LLM-Governed):**
- ANY message containing a price, discount, payment term, or contractual commitment MUST pass through approval queue
- Gate classification is rule-based (regex + structured field detection), NOT LLM-based
- Approval requires explicit operator action (tap/click confirm) — no auto-approval, no timeout-based approval
- Rejected items return to draft state with operator annotations
- All gate decisions are logged with operator ID, timestamp, and action taken

**Escalation Policy:**
- Quotes above configurable threshold (default: $5,000) require manager-level approval
- Custom terms not matching any template require manager-level approval
- Two-strike rule: if an operator rejects the same assistant-drafted quote twice, the event is flagged for manual handling

## LLM Usage Policy (Deterministic Boundaries)

| Layer | LLM Allowed? | Purpose | Boundary |
|-------|-------------|---------|----------|
| Intent classification | Yes | Map client message to action code | Confidence threshold 0.85; below = escalate |
| Data retrieval | No | Fetch prices, dates, availability | Deterministic DB queries only |
| Response drafting | Yes (constrained) | Natural language around verified data | Template-fill only; no novel claims |
| Pricing generation | No | Calculate quotes | Rule-based pricing engine only |
| Approval decisions | No | Accept/reject commitments | Human-only |
| Summarization (internal) | Yes | Summarize event history for operators | Internal-only; never client-facing |

**Invariant:** No LLM output reaches a client without passing through either (a) a template-constrained pipeline with verified data injection, or (b) explicit human approval. There is no path where raw LLM generation reaches clients.

## Unknowns & Evidence Gaps

| ID | Unknown | Impact | Mitigation Plan |
|----|---------|--------|------------------|
| U-1 | Client adoption rate of WhatsApp channel vs. existing email/phone workflows | Determines ROI timeline; low adoption = low value | Run 2-week opt-in pilot with 3 partner firms; measure message volume shift |
| U-2 | Intent classification accuracy on real event-industry queries | Below-threshold accuracy = excessive escalations, negating automation value | Collect 500+ real client messages from partners pre-launch; benchmark classifier before go-live |
| U-3 | Operator trust in AI-drafted responses | Low trust = operators re-checking everything, eliminating time savings | Measure approval-without-edit rate during pilot; target ≥60% |
| U-4 | WhatsApp Business API rate limits and compliance requirements per market | Could constrain message volume or require template pre-approval from Meta | Audit Meta's current policies for target markets before development |
| U-5 | Latency tolerance for WhatsApp responses | Clients may expect <30s responses; LLM + data lookup pipeline may exceed this | Benchmark end-to-end latency; set SLA of <15s for template responses, <60s for escalation acknowledgment |
| U-6 | Pricing engine complexity across different event types | Simple per-unit pricing vs. complex multi-vendor bundled quotes may require different approval workflows | Interview 5 operators on their quoting process; categorize complexity tiers |

## Pass/Fail Readiness

### Pass Criteria (ALL must be met for Phase 1 launch):
- [ ] Intent classifier achieves ≥85% accuracy on held-out test set of real client messages
- [ ] Zero pricing/commitment leakage in 1,000-message simulation (no unapproved financial data reaches client)
- [ ] End-to-end response latency <15s for 95th percentile of template responses
- [ ] Approval gate correctly intercepts 100% of pricing-containing messages in adversarial testing
- [ ] At least 2 pilot firms complete 4-week trial with ≥70% of routine queries handled by assistant
- [ ] Operator NPS ≥30 (adjusted down from success target to reflect minimum viability)

### Fail Criteria (ANY triggers halt-and-reassess):
- [ ] Pricing leakage detected in any test or pilot scenario
- [ ] Intent classifier accuracy <75% after fine-tuning on domain data
- [ ] Pilot firms report increased operator workload (assistant creates more work than it saves)
- [ ] WhatsApp Business API compliance review surfaces blocking requirements for target markets

---

## Technical Constraints

- **Data Residency:** All client data processed and stored in region-appropriate infrastructure; no client PII sent to LLM providers without data processing agreements in place
- **Audit Trail:** Every client-facing message must be traceable to its data sources, template, and approval chain within 2 business days of request
- **Rollback:** Assistant can be disabled per-client or globally within 5 minutes without affecting console functionality
- **LLM Provider Independence:** Intent classification and template-fill layers must be abstractable across providers (no provider-specific prompt engineering that prevents migration)
