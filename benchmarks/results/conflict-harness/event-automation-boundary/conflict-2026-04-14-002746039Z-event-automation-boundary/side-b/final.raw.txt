accountability for commercial commitments and high-risk decisions.\n\n### In scope\n- WhatsApp assistant for event intake, clarification questions, status updates, and handoff notifications.\n- eligibility rules, approval requirements, pricing-rule execution, and all external side effects.\n- LLM usage limited to natural-language understanding, summarization, draft generation, tone adaptation, and classification or extraction into predefined schemas.\n- Enforcement architecture that keeps the LLM isolated from external APIs and requires all model outputs to be parsed, validated, and accepted by application code before affecting state or customer communication.\n- Human approval required before any pricing commitment, discount, refund, venue deposit instruction, cancellation penalty, contract term, or bespoke commercial promise is sent to a customer.\n- Audit logging for LLM inputs, generated drafts, validation results, human approvals, final outbound messages, and rule-engine outputs.\n\n### Out of scope for Phase 1\n- Fully autonomous quote commitment.\n- Fully autonomous negotiation.\n- Contract execution or legally binding e-signature flows.\n- Payment collection and refund processing.\n- Vendor marketplace procurement.\n- Multi-channel support beyond WhatsApp and the internal console.\n- Custom event design generation beyond operational draft assistance.\n\n### Core user stories\n- As an event requester, I want to describe my event in WhatsApp so I do not need to fill out a long form before getting help.\n- As an event requester, I want timely clarification questions so the organizer can understand date, location, headcount, budget, dietary needs, equipment, and schedule constraints.\n- As an internal operator, I want a structured event brief generated from the WhatsApp conversation so I can review requirements quickly.\n- As an internal operator, I want the system to flag missing information and policy-sensitive requests so I know what to resolve before a quote is prepared.\n- As an approver, I want pricing drafts and commercial exceptions routed to me before the customer sees them so commitments remain controlled.\n- As an internal operator, I want a full audit trail of assistant suggestions, human decisions, validation results, and customer-facing messages so disputes can be investigated.\n\n### Enforcement architecture\nPhase 1 must implement the LLM boundary as a system design requirement, not as prompt guidance alone.\n\nThe system must contain these components:\n- WhatsApp channel adapter: receives and sends WhatsApp messages, but only through commands signed or authorized by the deterministic orchestrator.\n- Deterministic orchestrator: owns event records, state transitions, required-field logic, approval gates, command creation, command validation, and side-effect execution.\n- LLM worker: receives scoped conversation context and returns structured proposals only. It must not hold credentials for WhatsApp, pricing, payment, contract, CRM, calendar, or vendor APIs.\n- Pricing service or pricing rules module: calculates price components using configured deterministic rules, versioned rate tables, or human-entered values. The LLM must not calculate or invent prices.\n- Approval service: records required approvals, approver identity, approval timestamp, approved text, pricing-rule version, and final sent message.\n- Audit log: stores source messages, model input references, model output, parser and validator outcome, state transition, command envelope, approval event, and external send result.\n\n### LLM boundary requirements\n- The LLM worker may return only one of the approved structured output types: `IntentClassification`, `FieldExtraction`, `EventBriefDraft`, `CustomerReplyDraft`, `RiskFlagSuggestion`, or `SummaryDraft`.\n- The LLM worker must not return executable commands. Application code must convert accepted structured proposals into typed commands only after validation.\n- The LLM worker must have no direct network route, credentials, SDK client, tool binding, or service account that can call WhatsApp send, pricing, payment, contract, cancellation, CRM update, or approval APIs.\n- All model outputs must pass schema validation, enum validation, confidence threshold checks where applicable, and workflow-state compatibility checks before the orchestrator may use them.\n- Any model output that fails validation, references a price not produced by the pricing service, attempts to make a commitment, or falls outside the current state must be rejected and routed to human review.\n- Prompt instructions may describe the boundary, but enforcement must come from service permissions, command validation, state-machine checks, and approval gates.\n\n### Typed command model\nOnly the deterministic orchestrator may create executable command envelopes. Phase 1 command envelopes must include `command_type`, `event_request_id`, `actor_type`, `source_reference`, `payload`, `requires_approval`, `approval_id`, `policy_checks`, `idempotency_key`, and `created_at`.\n\nAllowed Phase 1 command types:\n- `CreateEventRequest`\n- `UpdateEventField`\n- `CreateOperatorTask`\n- `RequestHumanApproval`\n- `SaveInternalSummary`\n- `SendApprovedWhatsAppMessage`\n- `MarkUnsupportedForHumanReview`\n\nCommands with customer-facing commercial content must have `requires_approval: true` and a valid `approval_id` before execution. The WhatsApp channel adapter must reject outbound sends unless the command type is `SendApprovedWhatsAppMessage` and the approval and policy checks are valid.\n\nDisallowed Phase 1 command behavior:\n- A model-originated command directly sends a WhatsApp message.\n- A model-originated value becomes a price, discount, refund, penalty, deposit instruction, or contract term.\n- Any service accepts free-form model text as authorization for a commercial commitment.\n- Any customer-facing commercial message is sent without a stored approval record.\n\n### Functional requirements\n\n#### WhatsApp assistant\n- Accept inbound customer messages through WhatsApp.\n- Identify whether the message is a new event request, follow-up, clarification answer, status request, cancellation request, complaint, or unsupported request using predefined categories.\n- Ask deterministic intake questions when required fields are missing: event date, location, attendee count, event type, budget range, service needs, contact identity, and deadline.\n- Generate draft customer replies using approved templates and conversation context.\n- Send only messages that pass schema validation, policy checks, workflow-state checks, and approval requirements.\n- Escalate to human review when the request includes pricing, discounts, refunds, penalties, contract terms, legal threats, safety concerns, VIP handling, ambiguous intent, or low-confidence classification.\n\n#### Internal console\n- Display all open event requests with status, owner, next action, risk flags, and SLA timer.\n- Show structured event brief extracted from conversation history.\n- Show source messages next to generated summaries and extracted fields.\n- Show missing fields and recommended next question.\n- Show draft quote components produced by deterministic pricing rules or explicitly entered by an authorized operator.\n- Require explicit human approval before any customer-facing price, discount, refund, penalty, contract term, or binding commitment is sent.\n- Let an operator edit outbound drafts before sending.\n- Record approval actor, timestamp, approved text, underlying rule output, rule version, policy-check result, and final sent message.\n- Provide search and filters by event date, requester, status, operator, and approval state.\n\n#### Deterministic workflow and LLM boundaries\n- Workflow state must be owned by application code through a finite state machine, not by the LLM.\n- Pricing calculations must be performed by deterministic pricing services, versioned rule tables, or human entry by authorized users, not by the LLM.\n- The LLM may draft a customer-facing explanation of an approved price but may not invent, alter, or commit to the price.\n- The LLM may classify requests only into predefined categories, with confidence thresholds and fallback to human review.\n- The LLM may summarize conversation history, but the console must retain source messages for verification.\n- Any action with external side effects must be represented as a typed command created by application logic and validated before execution.\n- The assistant must refuse or escalate when user intent falls outside known workflow states or required confidence thresholds.\n\n### Phase 1 workflow\n1. Customer messages the WhatsApp assistant.\n2. WhatsApp channel adapter stores inbound message and passes it to the deterministic orchestrator.\n3. Orchestrator creates or retrieves the event request record and determines the current workflow state.\n4. Orchestrator sends scoped context to the LLM worker for classification, extraction, summary, or draft generation.\n5. LLM worker returns a structured proposal, not an executable command.\n6. Orchestrator validates schema, enums, confidence, current-state compatibility, and policy flags.\n7. If intake data is missing and the response is low-risk, orchestrator creates a send command for the next approved intake question.\n8. If pricing, discounts, refunds, penalties, contract terms, ambiguous intent, or policy-sensitive content is detected, orchestrator creates a human-review task.\n9. Pricing service or authorized operator creates quote components.\n10. Any price, discount, refund, penalty, or commitment is routed to a human approver.\n11. Approval service records approver, approved text, price source, rule version, and approval timestamp.\n12. Orchestrator creates `SendApprovedWhatsAppMessage` only after policy checks and approval checks pass.\n13. WhatsApp channel adapter sends the approved message using the command idempotency key.\n14. Audit log records source message, model proposal, validation result, command, approval, and send result.\n\n### Non-functional requirements\n- Reliability: WhatsApp message ingestion and outbound sending should be idempotent to avoid duplicate customer messages.\n- Auditability: every automated draft, validation decision, approval decision, and final outbound message must be traceable.\n- Latency: assistant replies for low-risk intake questions should be fast enough to feel conversational; human approval flows may follow operator SLA.\n- Security: role-based access control for console users and approvers; service credentials must be limited to the components that need them.\n- Privacy: redact or restrict sensitive customer data according to internal data policies before sending scoped context to the LLM worker where feasible.\n- Observability: track message failures, classification confidence, validation failures, rejected model outputs, escalation rate, approval turnaround time, and sent-message error rate.\n\n### Success metrics\n- Intake completion rate for WhatsApp-originated event requests.\n- Reduction in operator time spent converting raw conversation into `SendApprovedWhatsAppMessage` with valid approval when commercial content is present.\n\n### Guardrail metrics\n- Unapproved commercial commitment incidents.\n- Incorrect price communicated to customer.\n- Duplicate outbound WhatsApp messages.\n- Unsupported request handled without escalation.\n- Operator override rate on LLM-generated drafts.\n- Rejected model-output rate by validation category.\n- Customer complaint rate about confusing or incorrect assistant responses.\n\n### Launch criteria\n- All pricing commitments require approval in test and production paths.\n- LLM worker has no credentials or direct API path to WhatsApp send, pricing, payment, contract, cancellation, CRM update, or approval APIs.\n- All LLM outputs are parsed into approved schemas and rejected when
